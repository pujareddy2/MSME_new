1. Repository Overview

This repository is a full-stack innovation platform portal. It supports user registration, team creation, problem statement browsing, application submission with file upload, AI-assisted evaluation, final judging, notifications, and role-based dashboards.

The frontend communicates with Spring Boot REST APIs, and the backend persists data in MySQL through JPA entities.

The implementation is designed for an innovation program workflow where a team leader forms a team, team members are provisioned, a solution is submitted, the submission is evaluated, and a judge finalizes the score.

Project purpose

  • Manage an innovation workflow from team leader registration to final judging.
  • Allow team leaders to form teams and submit solutions against published problem statements.
  • Allow evaluators and judges to review the same submission in stages.
  • Keep track of activities, submissions, and notifications in a structured way.

Problem solved

The platform replaces manual tracking of participants, teams, submissions, and review results. Instead of spreadsheets or offline coordination, the application centralizes the entire lifecycle into role-based pages and backend APIs.

Application type

It is a role-based platform management system for MSME-style innovation or competition workflows.

3. Role Analysis: Team Leader / Admin

A. Role purpose

Team Leader is the core participant role. This user registers, logs in, creates a team, adds members, browses problems, submits applications, and checks notifications and submission status.

Admin and Event Head are content-management roles. They create and publish problem statements and manage the challenge catalog.

B. Complete user flow

1. Registration. User opens the login page and chooses Register as Team Leader. The form collects name, email, phone, password, confirm password, team name, and college ID.
2. Client validation. The frontend checks that passwords match before sending the request.
3. API call. The frontend sends POST /api/auth/register/team-lead.
4. Backend processing. The backend checks required fields, unique email and phone, hashes the password using BCrypt, and creates a user with role TEAM_LEAD.
5. Response. The backend returns an AuthResponse object with the profile data.
6. Session and navigation. The frontend stores session data in localStorage and redirects to team creation.
Login. User enters email or phone, password, and role on the login page. The frontend blocks empty submissions, then sends POST /api/auth/login. The backend searches by email or phone, checks that the selected role matches the stored role, and verifies the password hash. On success, the frontend stores the returned profile and routes the user to the correct dashboard.
Team creation. After registration or login, the team leader opens the Create Team page. The page allows a team name plus a table-style form for up to five members including the leader column. Member fields are name, email, mobile, gender, college, course, and roll number.
Validation. The frontend validates email format, mobile number length, required fields, duplicate emails, and duplicate roll numbers.
API call. The frontend sends POST /api/teams with leaderId, teamName, and the list of members.
Backend processing. The backend verifies leader role, one team per leader, unique team name, and valid member data. It creates the team, creates member user accounts with role TEAM_MEMBER, creates team member rows, and generates notifications.
Result. The frontend stores the created team and returns to dashboard.
Problem browsing. The team leader opens the problem statements page. The frontend fetches problems from GET /api/problems and application counts from GET /api/applications. Search, filters, and row limits are applied on the client side.
Application submission. The leader opens a problem and reaches the application form. The form collects abstract text, technology stack, GitHub link, demo link, and a file upload. The file must be PPT, PPTX, or PDF and within the 20 MB limit.
Submission request. The frontend sends multipart POST /api/applications.
Backend processing. The backend checks duplicate team/problem application, validates file type and size, stores the upload in uploads/applications, inserts the application row, and creates a notification.
Admin problem creation. Admin or Event Head logs in, enters problem ID, title, category, theme, deadline, and description, and sends POST /api/problems. The backend checks duplicate problem ID and inserts the new problem statement.

4. Role Analysis: Team Member / Participant

A. Role purpose

The team member role exists so the team leader can provision additional participants into the same team. The member can view team information, profile data, notifications, and team applications, but cannot create teams or submit applications.

B. Complete user flow

The team leader creates the team.
Backend automatically creates a TEAM_MEMBER user for each invited person.
The member logs in with email or phone, password, and role TEAM_MEMBER.
Frontend sends the shared POST /api/auth/login request.
Backend checks the role and password hash.
Frontend routes the member to the team-member dashboard or role dashboard.
Dashboard loads team information using GET /api/team-member/team/{userId}.
Dashboard loads applications using GET /api/team-member/applications?teamId=....
Profile page uses GET /api/profile/{userId} and may also show the team and recent applications.
Problem browsing is allowed, but application submission is blocked because only TEAM_LEAD can apply.

5. Role Analysis: Evaluator / Judge

A. Role purpose

The evaluator is the first review stage. It examines the submission, may generate AI guidance, and stores AI and manual scoring. The judge is the final stage and stores the final judge score after the evaluation stage has already completed.

B. Complete user flow

Evaluator login. Log in with role EVALUATOR and route to /evaluator.
Evaluator dashboard. Dashboard fetches applications from GET /api/applications and shows Pending or Evaluated status.
Open work item. Open /evaluate/{id}.
Load application. Page loads the application, abstract, and submission preview.
AI guidance. Page may call the external LLM wrapper to generate AI score and remarks.
Manual review. Enter manual score and manual remarks.
Submit evaluation. Send PUT /api/applications/{id}/evaluation.
Backend checks. Backend checks score ranges, required remarks, and duplicate evaluation prevention.
Persist results. Backend stores aiScore, aiRemarks, manualScore, manualRemarks, and marks the application EVALUATED.
Judge login. Log in with role JUDGE and route to /judge-dashboard.
Judge dashboard. Dashboard shows only applications that are evaluated or already judged.
Open judge workspace. Open /feedback/{id}.
Judge review. Page loads the application, AI output, evaluator output, and submission preview.
Final score. Enter judge score between 0 and 100.
Submit judging. Send PUT /api/applications/{id}/judging.
Backend checks. Backend checks that the application is in the correct stage and prevents repeat judging.
Persist results. Backend stores judge score, judgedBy, and marks the application JUDGED.
Notify leader. Team leader receives a notification.

F. Internal workflow logic

Evaluator page optionally calls AI service for structured scoring guidance.
Backend stores evaluator fields directly in the application row.
Judge stage reuses the same row and appends final judge score.

Main service behavior

ServiceCore rules
AuthServiceValidates registration data, hashes passwords, checks role match on login, supports password reset, builds AuthResponse.
TeamServiceEnsures only TEAM_LEAD can create a team, one team per leader, unique team name, valid member roster, creates member users and notifications.

ProblemStatementServiceHandles problem retrieval and creation with duplicate ID checks.
NotificationServiceCreates and lists notifications ordered by most recent.

Important backend behaviors

  • Password storage uses BCrypt hashes.
  • There is no JWT authentication flow.
  • CORS is enabled at controller level with wildcard origins.
  • Application file upload is limited to PPT, PPTX, or PDF up to 20 MB.
  • Applications are written directly into the applications table; evaluation and judging data are stored in the same row.
  • Default problem statements are seeded at startup by DataInitializer.

9. AI and External Integration

The evaluator page can generate an AI suggestion by calling an external chat-completion endpoint from frontend/src/utils/gemini.js. The request uses a system prompt and the application abstract/problem context. The returned JSON-like response is parsed for score and remarks. If the API fails, the UI falls back to a locally generated score suggestion.

This means AI assistance is currently a frontend responsibility, not a backend service responsibility.

10. Overall System Flow

Team leader registers or logs in.
Team leader creates the team and adds members.
Admin or Event Head publishes problem statements.
Team leader browses problems and submits the application.
Evaluator reviews the application and writes AI/manual scores.
Judge reviews the evaluated application and writes the final judge score.
Notifications and reports show the result of each stage.

Full lifecycle

Registration -> Team creation -> Problem selection -> Application submission -> Evaluation -> Judging -> Report display -> Notification feed.

12. Key Strengths, Missing Components, and Suggestions

Strengths

  • Clear controller-service-repository layering.
  • Complete role-based workflow coverage across leader, member, evaluator, and judge.
  • Practical duplicate checks and file validation.
  • Environment-driven frontend API URL support for deployment flexibility.
  • Useful notifications and dashboards for day-to-day monitoring.

Missing or incomplete components

  • No JWT or server-side session authorization layer is present.
  • The normalized evaluation tables exist but are not the primary runtime path.
  • Leaderboard is based on submission count, not final performance.
  • Automated tests are minimal.
  • CORS is permissive and should be tightened for production.

Suggestions for improvement

  1. Add JWT-based authentication and backend authorization guards.
  2. Move AI scoring into the backend to protect API keys and centralize behavior.
  3. Use Flyway or Liquibase for migration control.
  4. Normalize evaluation and judging data if that is the intended design.
  5. Add integration tests for registration, team creation, submission, evaluation, and judging.
  6. Tighten production CORS and add stronger validation annotations.
